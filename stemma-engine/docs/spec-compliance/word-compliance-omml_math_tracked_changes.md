# Word-compliance sweep — Math under tracked changes: m:ins/m:del control characters + math in inserted/deleted runs

0 confirmed gaps, 12 new regression tests (green), 1 test-bug discarded. Build status: green — `cargo test -p stemma --test spec_omml_math_tracked_changes_word_compliance` reports 12 passed, 0 failed, 0 ignored.

## Confirmed incompliances

None. The sweep probed the two distinct ways tracked changes meet OMML — (1) math-control-character revisions inside `m:ctrlPr` (`CT_MathCtrlDel` / `CT_MathCtrlIns`, ECMA-376 §17.13.5.{13,16}, §22.1.2.23, EG_RPrMath, MS-OI29500 §2.1.329 / §2.1.332), where the `w:del`/`w:ins` carries a current `w:rPr` (never `delText`, never `rPrChange`) and must round-trip verbatim as a direct child of `m:ctrlPr` rather than being hoisted to run level or flagged by the run-level `CT_RunTrackChange` content model; and (2) whole inline equations wrapped in run-level `w:ins`/`w:del` (§17.13.5.14 / §17.13.5.18), where accept/reject must keep-or-drop the entire `oMath` as a single U+FFFC opaque anchor — and stemma matched Word on every case (opens-clean, verbatim preservation, and accept/reject text). Ranking note: with zero pipeline-bugs and zero model-bugs found, there is nothing to rank.

## New regression tests

- `math_ctrl_del_inside_ctrlpr_not_consumed_as_run_deletion` — a `w:del` in `m:ctrlPr` is a control-character revision, not run-content deletion; accept and reject both leave exactly one U+FFFC (§17.13.5.13 / §2.1.329).
- `math_ctrl_del_roundtrips_verbatim_inside_ctrlpr` — the `m:ctrlPr><w:del` adjacency survives serialization verbatim, stays a direct child of `m:ctrlPr`, and never synthesizes a `w:delText` (§22.1.2.23 / §17.13.5.13).
- `inserted_inline_equation_wrapped_in_w_ins_kept_on_accept_dropped_on_reject` — accepting a `w:ins` run materializes the inline math (one U+FFFC); rejecting removes it entirely (§17.13.5.18).
- `deleted_inline_equation_wrapped_in_w_del_dropped_on_accept_kept_on_reject` — accepting a `w:del` run drops the whole inline equation; rejecting restores it (one U+FFFC) (§17.13.5.14).
- `math_ctrl_ins_with_rpr_child_no_rprchange_opens_clean_and_roundtrips` — a math-control `w:ins` carrying an (empty) `w:rPr` opens clean, keeps `m:ctrlPr><w:ins` adjacency, and never introduces an `rPrChange` (§17.13.5.16 / §2.1.332).
- `deleted_math_ctrl_char_in_ctrlpr_roundtrips_verbatim` — a `w:del` with `w:rPr` (Cambria Math) under `m:ctrlPr` preserves the wrapper plus the `w:del` author/date attributes and opens clean (§22.1.2.23 / §17.13.5.13 / §2.1.329).
- `deleted_math_ctrl_char_allows_current_rpr_opens_clean` — a current `w:rPr` (not `w:rPrChange`) inside a deleted-math-control `w:del` is valid markup Word opens without repair (§2.1.329 / §17.13.5.13).
- `inserted_math_ctrl_char_no_rpr_opens_clean` — an `rPr`-less `w:ins` under `m:ctrlPr` is valid, opens clean, and round-trips with its `w:id` intact (§2.1.332 / §17.13.5.16).
- `math_ctrl_del_with_rpr_opens_clean` — a deleted math control character with a current `w:rPr` inside an `m:oMathPara` validates without flags (§2.1.329 / §22.1.2.23 / CT_MathCtrlDel).
- `math_ctrl_del_preserved_verbatim_in_ctrlpr` — a `w:del` under the `m:ctrlPr` of an `m:nary` is re-emitted as a direct `m:ctrlPr` child with its author, not hoisted to run level, and opens clean (EG_RPrMath / §22.1.2.23).
- `math_ctrl_ins_without_rpr_opens_clean` — an empty `w:ins` under `m:ctrlPr` validates with no errors; the `rPr` child is not required (§2.1.332 / CT_MathCtrlIns).
- `math_ctrl_del_inside_omath_not_run_track_content_model` — a `w:del` under `m:ctrlPr` containing `w:rPr` is valid per `CT_MathCtrlDel`; the run-level `CT_RunTrackChange` content-model check is suppressed inside OMML (§17.13.5.13 / §22.1.2.23).

## Discarded test-bugs

- `display_math_omathpara_in_wins_is_single_opaque_anchor` — the fixture was schema-invalid OOXML, so stemma's `parse()` correctly fails fast; the expectation was wrong, not the engine. The body was `<w:ins><w:r><m:oMathPara>...</m:oMathPara></w:r></w:ins>`, i.e. `m:oMathPara` nested inside a `w:r`. Per the normative schema, the content model of `CT_R` is `EG_RunInnerContent` (reference/schemas/transitional/xsd/wml.xsd), an exhaustive choice that contains no math element at all — not `oMathPara` and not even `oMath`. Office Math is reached at the run/paragraph level via `EG_MathContent` (= `oMathPara | oMath`) inside `EG_RunLevelElts` (wml.xsd), and a tracked display-math insert is reached via `CT_RunTrackChange = (EG_ContentRunContent | m:EG_OMathMathElements)*` (wml.xsd; RNC wml.rnc). `EG_ContentRunContent` includes `EG_RunLevelElts`, so the schema-valid markup is `<w:ins><m:oMathPara>...</m:oMathPara></w:ins>` — `oMathPara` as a DIRECT child of `w:ins`, never wrapped in `w:r`. The MS-OE376 note for `oMathPara` (reference/ms-implementer-notes/ms-oe376/1859_...7.1.2.78...md) documents only that `oMathPara` must occur inside a `p` and lists what Word allows as children OF `oMathPara` (incl. `r`); it gives no allowance for `oMathPara` as a child of `r`. Decisive contrast: the sibling passing test `inserted_inline_equation_wrapped_in_w_ins_kept_on_accept_dropped_on_reject` uses `<w:ins><w:r><m:oMath>` (inline `oMath`, which `is_run_widget` tolerates), while the bad test substituted `oMathPara` — a block element with no place inside a run. stemma already handles the correct form: word_ir.rs (the tracked-container child loop) treats `m:oMathPara` as a paragraph widget when it is a direct child of `w:ins`, with a comment literally citing "an inserted/deleted block equation is `w:ins > m:oMathPara`". Adding `"oMathPara"` to `is_run_widget` to accept the invalid nesting would be a silent fallback on invalid input — exactly what CLAUDE.md's prime directive forbids; the fix belongs in the test, at the source of the bad data. The function is not present in the current test file (which holds the 12 green regression tests above), so no deletion edit is needed.
